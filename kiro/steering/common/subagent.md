# サブエージェント運用

## 基本ルール

- prompt_template は必ず英語で書く（日本語はエスケープ問題で呼び出し失敗のリスクがある）
- ステージ名は汎用形式を使う: phase1_a, phase1_b, phase2, phase3_a など
- 実装（ファイル書き込み）はメインエージェントが行う。サブエージェントは調査・分析・評価に使う
- 並列エージェントがファイルを書き込む場合は、他エージェントの存在を prompt_template で伝える

## パイプライン設計

### 基本構造: Fan-out → Fan-in → Fan-out

精度を上げるための多段フィルタリング。広く集めて、統合して、評価する。

```text
Phase 1 (並列探索)        Phase 2 (集約)      Phase 3 (並列評価)
phase1_a ─┐                                  ┌─ phase3_a
phase1_b ─┤→ phase2 (1エージェント) →┤─ phase3_b
phase1_c ─┤                                  ├─ phase3_c
phase1_d ─┘                                  └─ phase3_d
```

- Phase 1: 役割・着眼点・目的で分割する（ファイルやモジュール単位ではない）
- Phase 2: 各意見を漏れなく構造化し、矛盾点は両論併記する
- Phase 3: 集約された案を役割別に評価する
- Phase 4（任意）: 最終集約。ロングランではエージェントを置く、軽ければメインで受け取る

### 簡易構造: Fan-out → Fan-in

評価フェーズが不要な場合。

```text
Phase 1 (並列探索)        Phase 2 (集約)
phase1_a ─┐
phase1_b ─┤→ phase2 or メインで受け取る
phase1_c ─┤
phase1_d ─┘
```

## 並列数の判断

デフォルトは4並列（重い方に寄せる）。

| タスク | 並列数 |
|--------|--------|
| プロジェクト全体の設計・調査 | 4 |
| 広範囲のリファクタリング調査 | 4 |
| 不具合の原因調査 | 4 |
| 局所的な設計判断 | 2 |
| 単純な確認・小さなタスク | 並列なし |

## 集約エージェント

- 原則として集約エージェントを置く
- 出力が小さい場合はメインエージェントが直接受け取ってもよい
- 集約エージェントには「全意見を漏れなく構造化し、矛盾点は両論併記する」ことを指示する

## 出力フォーマット

並列エージェントの出力は統一フォーマットで揃える。情報の欠落や視野の狭窄を起こさない程度の汎用的な構造とする。

```text
## Summary
(簡潔な要約)

## Findings
(発見事項を構造化して列挙)

## Concerns / Risks
(懸念点・リスク)

## Recommendations
(推奨事項)
```

## 評価フェーズの精度

- 評価エージェントには「実際にファイルを読んで検証せよ」と明示する
- 主張の裏付けなしに CONFIRMED/FALSE を判定させない
- 広い探索では誤検知が出る前提で、評価フェーズで潰す

## 再調査のタイミング

- 大きな変更セット完了後に再調査を検討する
- 「もう問題ない」と判断するまで調査→修正→再調査のループを回す
- 2回連続で「STOP」判定が出たら終了
